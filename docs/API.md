## 1. Overview

The audio-trial backend is an evaluation harness for AR-glass-style voice-
assistant components. It exposes a uniform HTTP + WebSocket API so an
external service can drive three flavours of inference without reading source:

- **Single-adapter inferences** — pick one adapter (ASR, TTS, speaker-verify,
  language-ID, intent LLM, dispatch) and run it against a stored audio clip.
- **Multi-stage recipe runs** — pick a pre-built pipeline (`asr-then-tts`,
  `slow-loop`, `slow-loop-routed`, …) and bind concrete adapters to the
  category placeholders; the runner threads outputs to inputs and streams
  per-stage events.
- **Realtime omni sessions** — open a bidirectional WebSocket, stream Int16
  PCM @ 16 kHz (and optionally JPEG video), receive interleaved
  text + 24 kHz PCM audio replies from a multimodal model.

All three flavours share the same adapter registry (`/api/adapters`), the
same clip store (`/api/clips`), and the same speaker-enrolment store
(`/api/enroll`).

## 2. Base URL & deployment

Three deployment lanes are available; pick whichever matches your access
pattern. They are all equivalent surfaces — the same FastAPI process serves
both `/api/*` and the SPA at `/`.

| Lane | URL | When to use |
|---|---|---|
| **Public (Cloudflare Tunnel)** | `https://<random>.trycloudflare.com` | External integrators / partner team. Ephemeral hostname per `run-public.sh` start; fetch the live one with `bash deploy/run-public.sh url`. No auth, no rate limit — treat as test-only. |
| **Same-host** | `http://localhost:8000` | Backend on the same box as your client (e.g. an evaluator script on the dev box). |
| **Through the SPA origin** | `https://<your-host>/api/...` | The frontend mounts at `/`; `/api/*` and `/ws/*` are routed to the backend on the same origin. |

The current public URL at the time of writing this doc was
`https://contracts-soonest-increase-careful.trycloudflare.com`. Cloudflare
rotates this on every restart of the tunnel — don't hard-code it.

## 3. Auth & rate limiting

**There is no auth.** No bearer header, no API key, no session cookie. Don't
waste time looking for one.

**There is no rate limit** at the trial-app layer. Cloud adapters
(Deepgram, AssemblyAI, OpenAI, …) do enforce their own per-key quotas — those
manifest as `503` from `/api/runs` if the upstream rejects the call.

The MVP is single-tenant by design: this is evaluation infrastructure, not a
customer product. If you put it behind any kind of public traffic, front it
with an auth proxy (Caddy basic-auth, an OAuth2 sidecar, etc.) before
exposing it.

## 4. Conventions

- **Request/response bodies:** JSON unless noted. Multipart only on the two
  upload routes (`POST /api/clips`, `POST /api/enroll`).
- **Timestamps:** ISO-8601 UTC with `Z` suffix (e.g. `2026-04-28T12:34:56.789Z`).
- **IDs:** opaque strings (currently UUIDv4 hex — `clip_…`, `run_…`,
  `profile_…`). Don't parse them.
- **Latencies:** wall-clock milliseconds as `float`.
- **Costs:** USD as `float`, or `null` when not estimable (most edge adapters).
- **CORS:** allows `localhost:3000`, `localhost:5173`, plus the value of
  `FRONTEND_ORIGIN` and `PUBLIC_ORIGIN` env vars. Credentials are allowed.
- **Run-flow shape:** synchronous for batch adapters (`is_streaming: false`
  in `/api/adapters`) — `POST /api/runs` blocks until done and returns the
  final `Run` with `result`. Streaming adapters (`is_streaming: true`) return
  a placeholder `Run` immediately and emit `StageStarted` →
  `StageProgress*` → `StageCompleted | StageFailed` over `/ws/run/{id}`.
  Either way, `GET /api/runs/{id}` always returns the latest known state.
- **WebSocket binary protocol:** `/ws/omni` and `/ws/mic` accept raw binary
  frames; `/ws/run/{id}` is JSON-only.
- **NDJSON streaming:** the model-server side (out of scope here, but
  reachable on port 9101) speaks NDJSON for omni; the trial-app multiplexes
  that into WebSocket JSON events for the browser.

### 5.1 Discovery

#### `GET /api/health`
Liveness probe.

```bash
curl https://contracts-soonest-increase-careful.trycloudflare.com/api/health
# → {"status":"ok","version":"0.1.0"}
```

#### `GET /api/adapters?category=<cat>`
Returns the registered adapter list. Optional `?category=` filter
(`asr | tts | speaker_verify | lid | intent_llm | dispatch | realtime_omni`).
Each entry carries its `id`, `category`, `display_name`, `hosting`, `vendor`,
typed `inputs` / `outputs`, JSON-Schema `config_schema`, an optional
`cost_per_call_estimate_usd`, and an `is_streaming` flag. **This is the
canonical discovery endpoint** — never hard-code adapter IDs in your
integration; pull them from here.

```bash
curl http://localhost:8000/api/adapters | jq '.[0]'
# → {
#   "id": "deepgram",
#   "category": "asr",
#   "display_name": "Deepgram Nova-3",
#   "hosting": "cloud", "vendor": "Deepgram",
#   "inputs":  [{"name":"audio","type":"audio_file"}],
#   "outputs": [{"name":"text","type":"text"},{"name":"words","type":"word_timing"}],
#   "config_schema": {"type":"object","properties":{...}},
#   "cost_per_call_estimate_usd": null,
#   "is_streaming": true
# }
```

#### `GET /api/recipes` and `GET /api/recipes/{id}`
List or fetch built-in pipeline recipes. A recipe is a `stages[]` + `edges[]`
graph where each stage carries a `category` placeholder; the caller resolves
the placeholder to a concrete adapter at run time. See §8 for the full
inventory.

#### `GET /api/settings`
Read-only summary of which API-key env vars and service URLs are configured
on the running backend. **No secret values are returned** — only the strings
`set` / `unset` for each key, plus the bare URL (no token) for service URLs.
Use this to drive a `Settings` UI that shows green/grey "configured" chips.

### 5.2 Clips

Clips are uploaded once and referenced by `clip_id` in every later run. The
backend normalises uploaded video to wav-16k-mono via ffmpeg; adapters always
read the canonical wav.

#### `POST /api/clips`
Multipart upload. Form fields:

| Field | Type | Required | Notes |
|---|---|---|---|
| `file` | file | yes | Any container ffmpeg can read. Video tracks are stripped. |
| `source` | string | no | Default `"upload"`. Other values: `"record"`, `"live-mic"`. |
| `uploaded_by` | string | no | Free-form user tag. |

Returns `201` with the created `ClipOut` (id, duration_s, sample_rate,
channels, format, etc.). `400` on unreadable container; `500` if ffmpeg is
missing.

```bash
curl -F file=@meeting.wav -F source=upload \
     http://localhost:8000/api/clips
```

#### `GET /api/clips`
Returns `{"clips": [ClipOut, …]}`. No pagination yet — the corpus is small.

#### `PATCH /api/clips/{clip_id}`
JSON body, all fields optional:
`user_tags`, `scenarios`, `language_detected`, `snr_db`,
`speaker_count_estimate`, `captured_transcript`,
`captured_transcript_segments`. Used by the Corpus page chip-tagger and by
the auto-tagger. Other fields (`id`, `format`, `duration_s`, …) are
immutable post-ingest by design.

#### `DELETE /api/clips/{clip_id}`
Removes the clip directory under `DATA_DIR/clips/<id>/`. `204` on success,
`404` if not found.

#### `GET /api/clips/{clip_id}/audio`
Streams the canonical audio file with the matching `Content-Type`
(`audio/wav`, `audio/mpeg`, `audio/ogg`, `audio/flac`, `audio/mp4`). Useful
for previewing a clip from a partner UI.

#### `POST /api/clips/{clip_id}/autotag` and `POST /api/clips/autotag-all`
Run the heuristic auto-tagger (SNR, spectral centroid, optional Whisper-LID
+ Resemblyzer) on one or all clips. JSON body (optional):
`{ use_lid: bool=true, use_speaker_spread: bool=true, replace: bool=false }`.
Returns the detected scenarios, the merged final scenario list, the raw
feature values, and per-tag evidence strings.

### 5.3 Runs

#### `POST /api/runs`
Start a single-adapter run. JSON body:

```json
{ "clip_id": "<id>", "adapter": "<id>", "config": { /* per-adapter knobs */ } }
```

Returns `RunOut` with `id`, `clip_id`, `adapter`, `started_at`, `finished_at`,
`latency_ms`, `cost_usd`, `input_preview`, `output_preview`, `result`,
`error`. **Synchronous adapters** (most TTS, speaker-verify, LID, intent_llm,
dispatch — `is_streaming: false`) block until the run completes, so the
response already contains `result`. **Streaming adapters** (`is_streaming:
true` — most ASR + the omni proxy lane) return immediately with a placeholder
`Run`; the client should open `/ws/run/{id}` to watch partials.

`config` shape varies per adapter; pull it from `adapter.config_schema` in
`/api/adapters`. For example, Deepgram accepts
`{"language": "en", "model": "nova-3", "diarize": true}`.

```bash
curl -X POST http://localhost:8000/api/runs \
     -H "Content-Type: application/json" \
     -d '{"clip_id":"<id>","adapter":"parakeet","config":{"language":"en"}}'
```

#### `GET /api/runs/{run_id}`
Fetch a run record (live or persisted). Mid-stream runs are served from an
in-memory cache; finished runs are read from the JSONL store
(`DATA_DIR/runs/`). `404` if the id is unknown.

#### `POST /api/runs/recipe`
Run a multi-stage recipe. JSON body:

```json
{
  "clip_id": "<id>",
  "recipe_id": "asr-then-tts",
  "stage_adapters": { "asr": "parakeet", "tts": "cartesia_tts" },
  "stage_configs":  { "asr": {"language":"en"}, "tts": {"voice_id":"..."} }
}
```

`stage_adapters` is **mandatory** because recipes ship with `adapter: null`
placeholders. The runner walks `stages` in order, threads outputs to inputs
according to `edges`, and emits `RunStarted` / `StageStarted` /
`StageCompleted` / `StageFailed` / `RunFinished` events to
`/ws/run/{run_id}`. Returns the final `RecipeRunOut` with per-stage timings.

### 5.4 Recipes

Discovery only — see §5.1 (`GET /api/recipes`, `GET /api/recipes/{id}`) and

### 5.5 Enrollment

Speaker-verify adapters score audio against a pre-computed reference
embedding. The enrolment store lets you upload a reference clip once, then
reference the embedding by `profile_id` in every later run.

#### `POST /api/enroll`
Multipart upload. Form fields:

| Field | Type | Default |
|---|---|---|
| `file` | file (audio) | required |
| `adapter` | string | `pyannote_verify` (must be `category: speaker_verify`) |
| `profile_id` | string | `wearer` |

Returns `EnrollOut { profile_id, adapter, embedding_dim, embedding_dtype,
duration_s, saved_to }`. The embedding is persisted to
`DATA_DIR/enrollments/<profile_id>.json`.

#### `GET /api/enroll`
Lists every saved profile (`profile_id`, `adapter`, `embedding_dim`,
`saved_to`, `enrolled_at`). The raw embedding is **not** in this list — use
the next route.

#### `GET /api/enroll/{profile_id}/embedding`
Returns `{ profile_id, adapter, embedding_b64, embedding_dim }`. The slow-
loop runner uses this to feed `verify_segments()` without round-tripping
through the client.

#### `DELETE /api/enroll/{profile_id}`
`204` on success, even if the profile was already absent.

## 6. WebSocket endpoints

WebSockets are NOT in `openapi.json` (OpenAPI 3.0 doesn't standardise them),
so this section is the canonical reference.

### 6.1 `/ws/run/{run_id}` — streaming run events

JSON-only. The client connects after `POST /api/runs` (or
`POST /api/runs/recipe`) returns. Events emitted before the WS connects are
buffered and replayed on first connect; thereafter they are pushed live.

The client may send `"ping"` (text) and will receive `{"event":"pong"}`. A
`{"event":"keepalive"}` is pushed every 60 s of silence to defeat proxy
timeouts.

Event shapes (every event also carries `run_id` and `timestamp`):

```jsonc
// Single-adapter runs
{ "event": "StageStarted",   "adapter": "parakeet" }
{ "event": "StageProgress",  "adapter": "parakeet", "partial_text": "...", "partial_index": 7 }
{ "event": "StagePartial",   "adapter": "parakeet", "text": "..." }   // batch adapters only
{ "event": "StageCompleted", "adapter": "parakeet", "latency_ms": 432.1, "cost_usd": null,
                             "result": { "text": "...", "words": [...], "..." }, "is_streaming": true }
{ "event": "StageFailed",    "adapter": "parakeet", "error": "RuntimeError: ..." }

// Recipe runs add framing events
{ "event": "RunStarted",  "recipe_id": "asr-then-tts" }
{ "event": "RunFinished", "total_latency_ms": 1234.5, "total_cost_usd": 0.0, "error": null }
```

### 6.2 `/ws/omni` — realtime omni session

Bidirectional. Query parameters:

| Param | Required | Default | Notes |
|---|---|---|---|
| `adapter` | yes | — | Adapter id; must have `category: realtime_omni` (today: `minicpm_o`). |
| `profile_id` | no | `wearer` | Used by the optional pyannote heartbeat to inject "wearer just said …" context lines. |

**Client → server: tagged binary frames.** Every frame is `[type:u8][payload:bytes]`:

| Tag | Payload | Meaning |
|---|---|---|
| `0x01` | Int16 LE PCM @ 16 kHz mono | Audio chunk |
| `0x02` | JPEG bytes | Video frame (camera) |
| `0x03` | empty | End-of-utterance / flush |

JSON text frames are also accepted as a control channel:
`{"event":"ping"}`, `{"event":"flush"}`, `{"event":"interrupt"}` (user
started speaking mid-reply — abort the in-flight stream),
`{"event":"stop"}` (close session).

**Server → client: JSON events.** Every event carries `server_ts_ms`.

```jsonc
{ "event": "OmniReady", "adapter": "minicpm_o", "wearer_enrolled": true,  "ts_ms": 1714... }
{ "type":  "transcript",  "text": "...", "is_final": false }
{ "type":  "text_delta",  "text": "the model's reply, token by token" }
{ "type":  "audio_b64",   "data": "<base64 PCM 24 kHz Int16 LE mono>", "sr": 24000 }
{ "type":  "meta",        "..." }
{ "type":  "done",        "latency_ms": 850.0, "cost_usd": null, "error": null }
{ "event": "WearerTag",   "user_segments": 4, "stranger_segments": 1, "n_segments": 5, "window_s": 6.0 }
{ "event": "OmniError",   "error": "..." }
```

The client should buffer `audio_b64` chunks and play them as a continuous
24 kHz Int16 mono PCM stream. `text_delta` and `audio_b64` interleave —
render text as it arrives.

### 6.3 `/ws/mic` — live mic capture

Browser-friendly streaming-ASR proxy. Wraps Deepgram-streaming and AssemblyAI
v3 streaming behind a uniform shape that mirrors `/ws/run/{id}` (so the
frontend can render the same partial-text UI for either vendor).

Query parameters:

| Param | Required | Default |
|---|---|---|
| `adapter` | yes | — | Today: `deepgram` or `assemblyai`. Must be `is_streaming: true`. |
| `sample_rate` | no | 16000 |
| `save` | no | `0` | `1` to persist the captured PCM as a corpus clip on stop, with the live transcript as a ground-truth seed. |

**Client → server:** raw `Int16 LE PCM` binary frames; `{"type":"stop"}` text
frame to end the session.

**Server → client:** same `StageStarted` / `StageProgress` / `StageCompleted`
/ `StageFailed` envelope as `/ws/run/{id}`. When `save=1` is set the
backend additionally emits `{"event":"ClipSaved","clip_id":"<id>"}` after
the run completes — that clip is now in `/api/clips` for downstream replay.

## 9. Quickstart code examples

### 9.1 Single-adapter ASR run

```bash
# 1. Upload a clip
CLIP=$(curl -s -F file=@meeting.wav http://localhost:8000/api/clips | jq -r .id)

# 2. Run Parakeet (streaming adapter; POST returns immediately)
RUN=$(curl -s -X POST http://localhost:8000/api/runs \
        -H "Content-Type: application/json" \
        -d "{\"clip_id\":\"$CLIP\",\"adapter\":\"parakeet\",\"config\":{\"language\":\"en\"}}" \
      | jq -r .id)

# 3. Poll the final state
curl -s http://localhost:8000/api/runs/$RUN | jq '.result.text'
```

```python
# Python: same flow, but stream partials over the WebSocket
import asyncio, json, httpx, websockets

BASE = "http://localhost:8000"
WSBASE = BASE.replace("http", "ws")

async def main():
    async with httpx.AsyncClient() as h:
        with open("meeting.wav", "rb") as f:
            clip = (await h.post(f"{BASE}/api/clips", files={"file": f})).json()
        run = (await h.post(f"{BASE}/api/runs", json={
            "clip_id": clip["id"], "adapter": "parakeet",
            "config": {"language": "en"},
        })).json()

    async with websockets.connect(f"{WSBASE}/ws/run/{run['id']}") as ws:
        while True:
            ev = json.loads(await ws.recv())
            if ev.get("event") == "StageProgress":
                print("partial:", ev["partial_text"])
            elif ev.get("event") in ("StageCompleted", "StageFailed"):
                print("done:", ev); break

asyncio.run(main())
```

### 9.2 Multi-stage recipe run (ASR → TTS round-trip)

```bash
curl -X POST http://localhost:8000/api/runs/recipe \
     -H "Content-Type: application/json" \
     -d '{
       "clip_id": "<clip-id>",
       "recipe_id": "asr-then-tts",
```

### 9.3 Realtime omni session

```python
# Push 16 kHz Int16 PCM with a 0x01 type tag; flush with 0x03; collect
# 24 kHz audio replies via base64 audio_b64 events.
import asyncio, base64, json, struct, wave
import numpy as np
import websockets

URL = "ws://localhost:8000/ws/omni?adapter=minicpm_o&profile_id=wearer"

async def main():
    pcm = open("hello.s16le16k.raw", "rb").read()  # raw Int16 LE 16 kHz mono
    chunks = [pcm[i:i+3200] for i in range(0, len(pcm), 3200)]   # 100 ms
    out_pcm_24k = bytearray()

    async with websockets.connect(URL) as ws:
        ready = json.loads(await ws.recv())
        assert ready["event"] == "OmniReady"

        async def push():
            for c in chunks:
                await ws.send(b"\x01" + c)               # tag 0x01 = audio
                await asyncio.sleep(0.1)                 # match real-time pacing
            await ws.send(b"\x03")                       # tag 0x03 = flush

        async def pull():
            async for raw in ws:
                ev = json.loads(raw)
                t = ev.get("type") or ev.get("event")
                if t == "text_delta":
                    print(ev["text"], end="", flush=True)
                elif t == "audio_b64":
                    out_pcm_24k.extend(base64.b64decode(ev["data"]))
                elif t == "done":
                    return

        await asyncio.gather(push(), pull())

    # Write the reply audio to disk for sanity-check
    with wave.open("reply.wav", "wb") as w:
        w.setnchannels(1); w.setsampwidth(2); w.setframerate(24000)
        w.writeframes(bytes(out_pcm_24k))

asyncio.run(main())
```

## 10. Error model

All `/api/*` errors are JSON: `{"detail": "<human-readable message>"}` (or
the FastAPI 422 validation envelope, `{"detail": [{"loc": ..., "msg": ...,
"type": ...}]}`).

| Code | Meaning in this API |
|---|---|
| `400` | Bad request — the adapter id is unknown, the wrong category, or the upload could not be transcoded by ffmpeg. |
| `404` | Clip / run / recipe / profile id not found. |
| `422` | FastAPI Pydantic validation error — your JSON body is missing fields or has the wrong type. |
| `500` | Internal error — an adapter raised, ffmpeg is missing, the auto-tagger crashed. The `detail` string usually identifies the failing component. |
| `503` | Most often: a self-host model server is offline. `parakeet` / `canary_*` need `MODEL_SERVER_URL` reachable; `qwen_intent` needs `INTENT_LLM_URL` reachable; `minicpm_o` needs `MINICPM_O_REALTIME_URL` reachable. Modal cold-starts can also briefly produce 503 — retry after ~30 s. |

WebSocket failures surface as `{"event":"StageFailed","error":"..."}` (or
`{"event":"OmniError",...}` for `/ws/omni`) rather than as HTTP status codes;
the socket is closed cleanly afterwards.

## 11. Versioning & stability

This is **v0**. There is no SemVer guarantee: breaking changes land on `main`
without a deprecation window. The OpenAPI spec snapshot at
`docs/openapi.json` is regenerated on demand and may shift between commits.

Things most likely to change without notice:
- Adapter ids and `config_schema` fields as new vendors are added.
- Recipe stage ids and edge ports as the runner evolves.
- WebSocket event names — especially `/ws/omni`, which is still being
  iterated alongside the omni adapters.

If you need a stable surface, pin against a specific commit SHA and
re-validate on each upgrade.
